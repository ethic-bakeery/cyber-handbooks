PHASE 01

Fundamentals & Architecture

What is Security Onion?

Security Onion is a free, open-source Linux distribution purpose-built for threat hunting, network security monitoring (NSM), and log management. It bundles enterprise-grade tools into a single deployable platform — a complete SOC-in-a-box.

DETECTION

Suricata fires signature-based alerts. Zeek generates rich protocol logs. Together they cover network visibility end-to-end.

STORAGE & SEARCH

Elasticsearch stores all logs. Logstash parses and enriches. Kibana visualizes. Every packet metadata is searchable in seconds.

RESPONSE

TheHive provides case management. Stenographer captures full PCAP. Analysts pivot from alert → evidence → conclusion in one platform.

// NETWORK ADAPTER ROLES

ens32 — MANAGEMENT

Has IP. Used for SSH, web UI access, updates. NEVER monitor on this interface — it creates feedback loops.

ens33 — MONITOR (sniffing)

No IP assigned. Promiscuous mode on. Receives SPAN/mirrored traffic. Suricata + Zeek listen here.

WHY: Separation of concerns

Mixing management + monitor on same NIC causes your own queries to appear as suspicious traffic. Keep them separate always.

// INSTALLATION WALKTHROUGH — STANDALONE MODE

1 — BOOT & START

Boot ISO. Select Install Security Onion. Create admin user. Choose STANDALONE deployment for single-node lab.

2 — INTERFACE SELECT

Assign management NIC (gets IP). Assign monitor NIC(s). Set HOME_NET — your internal subnet range (e.g. 192.168.1.0/24).

3 — SERVICES START

SO installs Elasticsearch, Logstash, Kibana, Suricata, Zeek, TheHive. Setup takes ~15–30 min. Do NOT reboot mid-setup.

PHASE 05

Log Analysis & Alert Investigation

// KIBANA — SEARCHING ALERTS

# Navigate: Kibana → Discover
# Index pattern: *:logstash-*

# KQL: All IDS alerts
event_type: "alert"

# Filter by severity (1=high)
alert.severity: 1

# Pivot on attacker IP
src_ip: "10.0.0.50" AND event_type: "alert"

# DNS queries containing suspicious strings
event_type: "dns" AND dns.query: *base64*

# Correlate: same src, different event types
source_ip: "10.0.0.50"

// ZEEK LOG FIELD ANATOMY

# conn.log — every connection
ts # timestamp
uid # unique conn ID (pivot key!)
id.orig_h # source IP
id.resp_h # destination IP
id.resp_p # destination port
proto # tcp/udp/icmp
service # detected application layer
duration # connection duration
orig_bytes# bytes from client
resp_bytes# bytes from server
conn_state# SF=normal, S0=no reply, REJ=reject

// CORRELATING EVENTS — THE UID PIVOT

The uid field in Zeek logs is the master correlation key. A Suricata alert, a conn.log entry, an HTTP request, and an SSL log for the same connection all share the same uid.

# Step 1: Find alert with uid
$ cat /nsm/suricata/eve.json | jq -r '.zeek_uid' | head -5
# Returns: CmAk7a3sQdEj1xyz

# Step 2: Pivot that uid across ALL Zeek logs
$ grep "CmAk7a3sQdEj1xyz" /nsm/zeek/logs/current/*.log
# Instantly see: DNS query → TCP conn → HTTP request for same session

# In Kibana:
zeek_uid: "CmAk7a3sQdEj1xyz"
# Reveals full session context across all log types

Detection Engineering in Security Onion

// SURICATA RULE MANAGEMENT IN SO

RULE DIRECTORIES

# Shipped rulesets (managed)
/opt/so/rules/suricata/

# YOUR custom rules go here
/opt/so/saltstack/local/
salt/suricata/rules/
# File: local.rules

# Reload rules (NEVER edit in /opt/so directly)
$ sudo so-suricata-reload

EXAMPLE CUSTOM RULES

# Detect DNS tunnel (long subdomain)
alert dns any any -> any any (
  msg:"SO-DNS Tunnel Query";
  dns.query;
  pcre:"/[a-z0-9]{40,}\./i";
  threshold:type both,
    track by_src,
    count 5, seconds 60;
  classtype:trojan-activity;
  sid:9100001;rev:1;)

// CUSTOM ZEEK SCRIPT — PORT SCAN DETECTION

# /opt/so/saltstack/local/salt/zeek/
# custom_scripts/portscan.zeek

module PortScan;

export {
  redef enum Notice::Type += {
    Scan
  };
}

global scan_table: table[addr]
  of set[port] &default=set();

event connection_attempt(c: connection) {
  add scan_table[c$id$orig_h][c$id$resp_p];
  if (|scan_table[c$id$orig_h]| > 50)
    NOTICE([$note=Scan,
      $msg="Port scan detected",
      $src=c$id$orig_h]);
}

DEPLOY ZEEK SCRIPT

# Apply via salt
$ sudo salt '*' state.apply zeek

# Restart Zeek to load script
$ sudo so-zeek-restart

# Test with PCAP replay
$ sudo so-import-pcap nmap_scan.pcap

# Check NOTICE log
$ cat /nsm/zeek/logs/current/notice.log
# Should see: PortScan::Scan notice

WHY SALT?

Security Onion uses SaltStack to manage configs across all nodes. Changes made through Salt propagate to all sensors automatically — this is how production SOCs push rule updates.

PHASE 07

Alert Validation & Rule Tuning

// SUPPRESS NOISY RULES

# Add to threshold.conf in Salt
# /opt/so/saltstack/local/salt/suricata/
# threshold.conf

# Suppress specific SID from internal scanner
suppress gen_id 1, sig_id 9100001,
  track by_src,
  ip 192.168.1.50/32

# Rate-limit noisy rule globally
threshold gen_id 1, sig_id 2019935,
  type limit, track by_src,
  count 1, seconds 300

# Reload after changes
$ sudo so-suricata-reload

// TUNING WORKFLOW

1

Identify noise: Sort alerts by count. Top 10 noisiest rules = tune targets. In Kibana: aggregation on alert.signature_id

2

Check source: Is it an authorized scanner (Nessus/Qualys)? Known monitoring tool? Add suppress for that IP.

3

Validate TP rate: Run rule against known-attack PCAP. If it still fires → keep rule, tune source whitelist only.

4

Document changes: Every suppression needs a ticket. Undocumented suppressions create blind spots that attackers exploit.

PHASE 08

Threat Hunting with Security Onion

// HUNT — BEACONING DETECTION

# Find hosts with regular-interval outbound conns
# Zeek: extract connection intervals per dst
$ zeek-cut id.orig_h id.resp_h ts \
  < /nsm/zeek/logs/current/conn.log \
  | awk '{print $1,$2,int($3/300)}' \
  | sort | uniq -c | sort -rn \
  | head -20

# In Kibana: date histogram on
# src+dst pair — look for regular spikes
id.orig_h: "10.0.0.5" AND id.resp_h: "8.8.8.8"

// HUNT — DNS EXFILTRATION

# Find unusually long DNS queries
$ zeek-cut query \
  < /nsm/zeek/logs/current/dns.log \
  | awk 'length > 50' \
  | sort | uniq -c | sort -rn

# High-entropy subdomains = Base64 data
$ zeek-cut query \
  < dns.log \
  | grep -E '[a-zA-Z0-9+/]{25,}\.'

# In Kibana KQL:
event_type: "dns" AND
dns.query.length: {50 TO *}

// SO HUNT — QUERYING STENOGRAPHER FOR FULL PCAP

# Pull full PCAP for specific IP + timeframe (Stenographer)
$ sudo so-pcap-export \
  --start "2024-01-15T14:00:00" \
  --end "2024-01-15T15:00:00" \
  --host 10.0.0.50 \
  --output /tmp/suspect_traffic.pcap

# Then analyze in Wireshark or tcpdump
$ tcpdump -r /tmp/suspect_traffic.pcap -n -v
# Or: wireshark /tmp/suspect_traffic.pcap
PHASE 09

Incident Investigation Workflow

SECURITY ONION ALERT — SEVERITY 1 — CRITICAL
RULE:
ET MALWARE Possible Reverse Shell
SRC:
10.0.0.55:50321
DST:
185.220.101.42:4444
ZEEK UID:
CmFk3a1sXdEj2abc
TIMESTAMP:
2024-01-15T14:32:11Z
DATA TRANSFERRED:
orig: 2.1MB resp: 450KB
1

TRIAGE — Confirm alert legitimacy

Check src host 10.0.0.55 in asset inventory. Is it a server, workstation, or monitored endpoint? Check if dst IP 185.220.101.42 is in threat intel feeds. Port 4444 = classic reverse shell port.

# Quick triage queries in SO Console
src_ip: "10.0.0.55" | timeframe: last 24h
# How many other alerts from this host?
2

INVESTIGATION — UID pivot + full context

# Pivot on Zeek UID to get full session
zeek_uid: "CmFk3a1sXdEj2abc"
# Find conn.log: duration? bytes? conn_state?

# Check what happened before the alert
id.orig_h: "10.0.0.55" AND @timestamp:[now-1h TO now]
# Recon? Downloads? HTTP requests?

# Pull full PCAP of the session
$ sudo so-pcap-export --uid CmFk3a1sXdEj2abc
3

SCOPE — How far did it spread?

Did 10.0.0.55 connect to other internal hosts after the reverse shell? Check lateral movement: new SSH connections, SMB activity, unusual auth events. Pull Zeek conn.log for all connections FROM this host in the last 6 hours.

4

RESPONSE — Contain & document

Isolate 10.0.0.55 at switch/firewall. Create TheHive case with all IOCs: IP, UID, timestamps, PCAP path. Tag MITRE ATT&CK technique (T1059 - Command & Scripting Interpreter). Escalate to IR team.
