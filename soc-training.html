PHASE 02

PCAP Analysis Mastery

WHAT TO LOOK FOR

  • ▸ Duplicate ARP replies for same IP from different MACs
  • ▸ Gratuitous ARP broadcasts (unsolicited)
  • ▸ ARP reply without prior request (opcode=2, no matching opcode=1)
  • ▸ Gateway IP resolving to non-gateway MAC

WIRESHARK FILTERS

# All ARP traffic
arp

# Duplicate IPs (ARP poison indicator)
arp.duplicate-address-detected

# Gratuitous ARP
arp.isgratuitous == 1

# ARP replies only
arp.opcode == 2

TCPDUMP COMMANDS

$ tcpdump -r arp.pcap -n arp -v

# See MAC:IP mappings
$ tcpdump -r arp.pcap -n arp \
  | awk '{print $NF, $9}' \
  | sort | uniq -c | sort -rn

KEY FIELDS

arp.src.hw_mac — sender hardware address
arp.src.proto_ipv4 — sender IP address
arp.opcode — 1=request, 2=reply
eth.src — Ethernet source MAC

🎯 HANDS-ON EXERCISE

Load your ARP PCAP. Apply arp.duplicate-address-detected. Find which IP is being poisoned and which two MACs are claiming it. Document attacker MAC, victim IP, and timestamps. Write it in your analyst report.

TUNNEL INDICATORS

  • ▸ Unusually long subdomains (Base64-encoded payloads)
  • ▸ High query volume to single domain
  • ▸ TXT, NULL, CNAME record abuse
  • ▸ Query length > 100 chars — data exfil signal
  • ▸ Requests for non-existent domains (NXDOMAIN storms)
# DNS tunnel detection filters
dns.qry.name.len > 50
dns.qry.type == 16 # TXT records
dns && frame.len > 512
# High freq same domain
dns.qry.name contains "evilsite"

// DNS TUNNEL FLOW

SMURF ATTACK

Spoofed ICMP Echo Requests to broadcast address → amplified DoS

icmp.type == 8
# + broadcast dst
eth.dst == ff:ff:ff:ff:ff:ff
# + spoofed src
ip.src == victim_ip

ICMP TUNNEL

Data in payload field — look for non-standard payloads > 64 bytes

icmp && data.len > 64
icmp.type == 0 || 8
# Check payload bytes
icmp contains "HTTP"

FRAGMENTATION

Overlapping fragments to bypass IDS — check ip.flags.mf and offsets

ip.flags.mf == 1
ip.frag_offset > 0
# Tiny fragments
ip.len < 28

// NMAP SCAN TYPES — TCP FLAG PATTERNS

Scan Type Flags Set Wireshark Filter Purpose
SYN Scan SYN only tcp.flags == 0x002 Port discovery, stealthy
ACK Scan ACK only tcp.flags == 0x010 Firewall mapping
FIN Scan FIN only tcp.flags == 0x001 IDS evasion
NULL Scan No flags tcp.flags == 0x000 IDS evasion
Xmas Scan FIN+PSH+URG tcp.flags == 0x029 OS fingerprinting
# Detect port scan (multiple ports, single src)
$ tcpdump -r nmap.pcap 'tcp[tcpflags] == tcp-syn' \
  | awk '{print $3}' | cut -d. -f1-4 \
  | sort | uniq -c | sort -rn | head
PHASE 03

Detection Logic Framework

// THE DETECTION ENGINEER THOUGHT PROCESS

🔍
BEHAVIOR ANALYSIS

Identify the exact primitive behavior: packet flags, payload patterns, rate, direction. Don't detect "hacking" — detect specific observable network actions.

⚖️
BASELINES MATTER

What does normal DNS traffic look like? Normal ARP? Build baselines first. Anomalies only exist relative to normal.

🎯
THRESHOLDS

1 NULL-flag packet = curiosity. 500 in 10 seconds = port scan. Detection logic must encode thresholds, not just patterns.

🌐
CONTEXT AWARE

An ARP reply is normal. 50 unsolicited ARP replies from one host in 5s is not. Always add temporal and count context.

📊
FP vs FN TRADEOFF

Loose rules = alert fatigue. Tight rules = missed attacks. Learn to tune: start tight, widen only with evidence.

🔗
CHAIN DETECTION

Real attacks chain: recon → exploit → C2 → exfil. Detect the chain, not just one step. MITRE ATT&CK maps the sequence.

PHASE 04

Rule Writing — Snort · Suricata · Zeek

// SURICATA/SNORT RULE ANATOMY

alert tcp $EXTERNAL_NET any -> $HOME_NET any (
  msg:"NMAP SYN Scan Detected";
  flags:S,12;
  flow:stateless;
  threshold:type threshold, track by_src, count 30, seconds 10;
  classtype:attempted-recon;
  sid:9000001;
  rev:1;
)

ACTION

alert / drop / pass / log — what to do when matched

HEADER

Protocol, src/dst IP, src/dst port, direction arrow

OPTIONS

msg, content, flags, flow, pcre, threshold, sid

SID

Unique rule ID. Custom rules: 9000001+. Never duplicate.

SNORT ARP Poisoning Rule
alert arp any any -> any any (
  msg:"ARP Poison - Gratuitous";
  arp-op:2;
  sid:9000010;
)
SURICATA DNS Tunnel Rule
alert dns any any -> any any (
  msg:"DNS Tunnel Long Query";
  dns.query;
  pcre:"/^[a-z0-9]{40,}/i";
  threshold:type both,
    track by_src,
    count 10, seconds 60;
  sid:9000020;
)
ZEEK Port Scan Detection
global scan_table: table
  [addr] of set[port]
  &default = set();

event connection_attempt(c: connection) {
  add scan_table[c$id$orig_h][c$id$resp_p];
  if (|scan_table[c$id$orig_h]| > 50)
    NOTICE([$note=Scan::PortScan,
      $msg="Port scan detected"]);
}

// COMPLETE NMAP SCAN DETECTION RULESET (Suricata)

# SYN Scan
alert tcp $EXT any -> $HOME any (
  msg:"NMAP SYN Scan";
  flags:S,12;
  threshold:type threshold,
    track by_src,
    count 30, seconds 10;
  sid:9000100; rev:1;)

# NULL Scan (no flags)
alert tcp any any -> any any (
  msg:"NMAP NULL Scan";
  flags:0;
  sid:9000101; rev:1;)
# Xmas Scan (FIN+PSH+URG)
alert tcp any any -> any any (
  msg:"NMAP Xmas Scan";
  flags:FPU,12;
  sid:9000102; rev:1;)

# ACK Scan (firewall mapping)
alert tcp $EXT any -> $HOME any (
  msg:"NMAP ACK Scan";
  flags:A,12;
  threshold:type threshold,
    track by_src, count 20, seconds 5;
  sid:9000103; rev:1;)

2

2

TRIAGE — Confirm it's real

Pull Zeek conn.log for src IP. How many unique dst IPs? What ports targeted? Any successful connections? Check DNS logs for C2 indicators. Check auth logs for lateral movement.

4

RESPONSE — Contain & remediate

Block src IP at firewall. Isolate host if compromised. Preserve evidence (pcap dump). Notify incident response team. Open ticket with full timeline and IOCs.
